# СЭМД-уведомления

Веб-сервис для статистов ГБУЗ ПК «Осинская ЦРБ»: загрузка отчётов РЭМД (ЕИСЗ ПК),
наглядная картина по подписанию/регистрации СЭМД и **рассылка «долгов» врачам по почте**.

Заточен под **Host Manager** (self-host панель): слушает `HOST`/`PORT` из окружения,
живёт на корне `/`, читает `X-Remote-User` / `X-Remote-Name` (с фиксом кодировки Latin-1→UTF-8).

## Возможности
- **Загрузка** выгрузок РЭМД (.xls = SpreadsheetML), автоопределение типа.
- **Дашборд**: воронка «сформировано → подписано → в РЭМД», KPI, топ-неподписантов, сводка ошибок.
- **Врачи / долги**: таблица по врачам, по кнопке — письмо каждому со списком его неподписанных документов.
- **Ошибки**: разбор ФЛК/регистрации (диагностика `OBJECT_NOT_FOUND` — маппинг подразделений).
- **FreeIPA**: подтягивание корпоративных почт по ФИО; ручное сопоставление.
- **Пакетная рассылка**: письма уходят одним SMTP-соединением с троттлингом (задержка между письмами + пауза пачками), в фоне — чтобы почта не блокировала за спам.
- **Ответственный за исправление**: документы без врача («Не указан»), ошибки ФЛК и затор на подписи МО уходят сводным отчётом ответственному (кнопка на странице «Ошибки»). Аналогичный отчёт по ФАП — на странице «ФАП».
- **История периодов**: каждая загрузка сохраняется по периоду; можно вернуться на ранее загруженный период и выгрузить сохранённый отчёт обратно (страница «Загрузка»).
- **Несколько почт на отделение**: в почте зав. отделением можно указать несколько адресов через запятую — письмо уйдёт всем.

## Поддерживаемые отчёты (минимальный набор)
| Файл (из ЕИСЗ ПК) | Тип | Для чего |
|---|---|---|
| Отчет по отправке документов в РЭМД в разрезе врачей | `vrachi` | рейтинг и % подписания |
| Список пациентов с неподписанными документами | `debts` | содержимое писем врачам |
| РЭМД. Детализация по ошибкам ФЛК | `flk` | ошибки регистрации |

Остальные отчёты распознаются, но пока не загружаются (легко добавить в `parser.py`).

## Запуск локально (для разработки)
```bash
python3 -m venv venv
venv/bin/pip install -r requirements.txt
PORT=5000 venv/bin/python app.py
# http://127.0.0.1:5000
```

## Деплой в Host Manager
1. Имя приложения — **ASCII** (например `semd-notify`), код в `/root/apps/semd-notify`.
2. Git clone / архив / вручную → панель сама создаст venv и поставит `requirements.txt`.
3. Задать переменные окружения (см. `.env.example`): SMTP_*, IPA_* . `PORT`/`HOST` подставит панель.
4. Назначить внешний порт → «Включить». Single Sign-On по cookie панели.

## Настройки (env или UI)
Параметры SMTP и FreeIPA задаются **двумя способами**:
- через **переменные окружения** в Host Manager (см. `.env.example`), либо
- на **странице «Настройки»** в самом приложении (сохраняются в БД, **приоритет над env**; пароли скрыты).

Ключевые: `SMTP_HOST/PORT/USER/PASS/FROM`, `SMTP_DRYRUN=1` (тест без отправки),
`SMTP_BATCH_DELAY=2` (сек между письмами), `SMTP_BATCH_SIZE=25` / `SMTP_BATCH_PAUSE=30` (пауза между пачками),
`IPA_LDAP_URI/IPA_BASE_DN/IPA_BIND_DN/IPA_BIND_PW`, `IPA_AUTOSYNC=1`, `IPA_SYNC_HOURS=24`,
`RESP_NAME` / `RESP_EMAIL` (ответственный за исправление — получатель сводного отчёта).

**Автосинхронизация FreeIPA**: при включённом `IPA_AUTOSYNC` фоновый поток раз в сутки
(`IPA_SYNC_HOURS`, проверка ежечасно) тянет корпоративные почты из IPA и сопоставляет врачам.
Время и результат последней синхронизации видны на странице «Настройки». Кнопка «Синхронизировать сейчас» — вручную.

## Защита персональных данных
Сервис обрабатывает **ПДн сотрудников** (ФИО, СНИЛС, почта) и **ПДн пациентов**
(в списках неподписанных — ФИО, № случая). Поэтому:
- только **внутренний контур** (за Host Manager с авторизацией), на 127.0.0.1;
- **в письмах — обезличенные данные**: только № случая, вид документа и период,
  **без ФИО и даты рождения пациента** (врач находит документ в ЕИСЗ ПК по номеру случая);
  ФИО пациентов отображается только в интерфейсе на внутреннем контуре;
- рассылка только на **корпоративные адреса `@med.permkrai.ru`**, каждому врачу — только его случаи;
- обработку зафиксировать в ОРД (основание — производственный контроль; приказ МЗ ПК № 34-01-03-601).

## Стек
Flask, SQLite (stdlib), ldap3. Парсинг — stdlib `xml.etree` (SpreadsheetML). Без внешних JS/CDN.

## Структура
```
app.py            Flask-приложение и роуты
parser.py         разбор отчётов РЭМД (SpreadsheetML)
storage.py        SQLite: данные, маппинг почт, журнал рассылки
mailer.py         SMTP-рассылка долгов
ipa.py            FreeIPA (LDAP) — корпоративные почты
templates/ static/  интерфейс
data/             БД и загруженные файлы (в .gitignore)
```

## MCP-сервер (доступ ИИ-ассистентов)

`mcp_server.py` — шина контекста и действий для нейронки: 25 инструментов трёх видов —
**чтение** (сводки по всем доменам, детализация ошибок, журналы), **изменение**
(тексты писем, теги и комментарии отчётов, получатели сводных, почты врачей и
заведующих, карта и планы стационаров), **отправка** (сводные отчёты ответственным;
по умолчанию выключена). Будущие модули (бухгалтерия, ЭДО, админ-утилиты) добавляют
свои инструменты в `TOOLS` — клиент обнаруживает их через `tools/list`.

- набор доступных инструментов управляется на странице «Настройки → MCP-инструменты»:
  выключенные не видны ассистенту и отклоняются при вызове (конфиг читается на лету);
- транспорт stdio (MCP 2024-11-05), без внешних зависимостей;
- подключение: `claude mcp add centr -- python /путь/до/mcp_server.py`
  (или аналогичная запись в конфиге любого MCP-клиента);
- ПДн: прямые идентификаторы пациентов (ФИО, СНИЛС) всегда вырезаются
  (`PATIENT_MASK`) — и из чтения, и из писем; врачи и сотрудники — служебный
  контекст, отдаются полностью;
- каждый вызов (включая отклонённые) пишется в журнал операций
  (страница «Журнал → Операции»); отправки — также в журнал рассылок.
